<!-- Research findings for API-02 (use/result.try) -->
After thorough analysis, zero candidates were found where use/result.try would improve readability:
- birch.gleam default_logger(): Fallback chain, not a pipeline. Uses lazy_or pattern.
- handler.gleam handle(): Error handling with side effects, not a pipeline.
- file.gleam: I/O error handling with fallbacks, not pipeline-style.
The codebase correctly uses `case` for control flow branching. No changes needed.

<!-- Research findings for API-04 (builder consistency) -->
All builder patterns already consistent:
- Logger: new(), with_level(), with_handler(), with_handlers(), with_context(), etc. -- all return Logger
- Handler: new(), with_min_level(), with_error_callback() -- all return Handler
- AsyncConfig: config(), with_queue_size(), with_flush_interval(), with_overflow() -- all return AsyncConfig
- JsonBuilder: builder(), add_field(), add_timestamp(), etc. -- all return JsonBuilder
No changes needed.
